2 ЗАВДАННЯ
Студент повинен обрати мову програмування, дослідити основні рекомендації для написання якісного коду у цій мові, продемонструвати ці рекомендації на прикладах коду, обґрунтувати кожну рекомендацію і надати детальні пояснення. Для кожного прикладу мають бути наведені фрагменти коду.
Кроки виконання:
1. Вибір мови програмування (узгодження з викладачем).
2. Дослідження рекомендацій (стиль, іменування, структура, рефакторинг, документація).
3. Підготовка прикладів коду (поганий/гарний код).
4. Пояснення кожної рекомендації (важливість, наслідки недотримання).
5. Підготовка презентації.
6. Завантаження результатів у репозиторій та оформлення звіту.
3 ОПИС ВИКОНАНОЇ РОБОТИ
В якості мови програмування для дослідження було обрано Swift. Ця мова, розроблена компанією Apple, має суворі вимоги до типізації та безпеки, а також чітко визначені офіційні гайдлайни (API Design Guidelines).
3.1 Структура коду та файлів 
Фундаментальним правилом організації проєкту на Swift є принцип: «Один тип — один файл». Назва файлу повинна ідентично відповідати назві оголошеного в ньому типу (наприклад, клас UserProfile має знаходитись у файлі UserProfile.swift). Файлова структура має базуватися на функціональних модулях (Modules/Features), а не лише на технічних шарах (Models, Views). Всередині файлів рекомендується використовувати директиву // MARK: - для логічного зонування коду (властивості, методи життєвого циклу, публічні методи, розширення).
3.2 Форматування 
Для забезпечення читабельності коду у Swift прийнято наступні стандарти форматування:
* Відступи: Використовується виключно 4 пробіли. Табуляція заборонена, оскільки вона може відображатися по-різному в різних IDE.
* Фігурні дужки: Дотримання стилю K&R (Kernighan and Ritchie), де відкриваюча дужка { знаходиться на тому ж рядку, що й оголошення функції або умовного оператора.
* Довжина рядка: Рекомендоване обмеження — 120 символів для уникнення горизонтальної прокрутки.
* Двокрапка: При оголошенні типів пробіл ставиться після двокрапки, але не перед нею (наприклад, let count: Int).
3.3 Іменування (Naming) 
Swift дотримується конвенції CamelCase:
* UpperCamelCase використовується для типів даних (class, struct, enum, protocol). Приклад: NetworkManager, UserProfile.
* lowerCamelCase використовується для екземплярів, властивостей, функцій та параметрів. Приклад: viewDidLoad, userName.
* Ясність: Назва повинна чітко пояснювати роль сутності. Абревіатури допускаються лише якщо вони є загальноприйнятими, скорочення на кшталт usr замість user є неприпустимими.
3.4 Специфічні конвенції мови
* Безпека (Safety): Для обробки опціональних значень перевага надається конструкції guard let для "раннього виходу" (early exit). Це дозволяє уникнути глибокої вкладеності коду ("піраміди смерті"), характерної для if let.
* Self: Використання ключового слова self має бути мінімізованим. Воно обов'язкове лише в ініціалізаторах (для розрізнення параметрів та властивостей) та в замиканнях (closures) для явного захоплення контексту.
* Розширення (Extensions): Рекомендується використовувати extension для винесення реалізації протоколів або групування приватних допоміжних методів, що робить основний клас чистішим.
3.5 Документування та інструменти 
Якісний код на Swift повинен містити документацію у форматі Markdown (починається з ///). Це дозволяє середовищу розробки Xcode автоматично генерувати підказки Quick Help. Для автоматичного контролю дотримання стилю використовується статичний аналізатор SwiftLint, який інтегрується у процес збірки проєкту.
4 ВИСНОВКИ
В ході виконання практичної роботи було проведено аналіз рекомендацій щодо оформлення коду мовою Swift. – Було розглянуто офіційні гайдлайни Apple (API Design Guidelines) та рекомендації спільноти Ray Wenderlich. – Отримано навички структурування Swift-проєктів, правильного іменування змінних та функцій, а також використання специфічних конструкцій мови (guard, extension) для покращення читабельності. – Встановлено, що стандартизація коду є критично важливою для командної розробки, оскільки вона знижує когнітивне навантаження при читанні коду, спрощує адаптацію нових розробників та зменшує кількість помилок завдяки автоматизованим інструментам, таким як SwiftLint.
